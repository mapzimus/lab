# Mobile + Accessibility Audit — 2026-05-25

Autonomous live-site audit performed via Chrome DevTools (Claude-in-Chrome MCP) against `https://maxwellhowegis.com/geopuesto/` and `https://maxwellhowegis.com/geopuesto/playground/` immediately after the 2026-05-24 split deploy.

## Methodology

- Browser: Chrome on Brontosaurus (Windows 11).
- Viewport: target was 390×844 (iPhone 12+ portrait) but Chrome enforced a minimum window width — measured viewport stayed at ~1707px CSS pixels even after the resize call succeeded. Findings below are at desktop viewport; touch-target measurements are CSS pixel sizes and apply at any viewport.
- Coverage: both consumer (`/geopuesto/`) and playground (`/geopuesto/playground/`) loaded fresh, probed via DOM introspection (`getBoundingClientRect`, computed styles, media-query enumeration).

## Consumer findings (`/geopuesto/`)

### WCAG 2.5.5 (target size) — gating mismatch

**Status**: 780 of 2,395 interactive elements measured below 44×44px CSS pixels. Most are SVG city markers, which are exempt from WCAG 2.5.5 under the "essential" clause. But the input-panel chrome itself falls short:

| Control | Class | Measured | Notes |
|---|---|---|---|
| `Quick Picks` tab | `.tab` | 112×32 | Mode toggle, top of input panel |
| `Search` tab | `.tab` | 78×32 | Mode toggle |
| `Coordinates` tab | `.tab` | 112×32 | Mode toggle |
| `📍 Use my location` | `.tab.tab-action` | 162×31 | Primary CTA in the input panel |
| `🎲 Shuffle` | `.tab-shuffle` | 98×31 | Quick-picks shuffle |
| `⇄ Swap` | `.swap-btn` | 80×33 | Swap origin/antipode |

Quick-pick preset cards (`.preset`) measure 195×62 — they pass.

### Root cause

`index.html` line 1572 opens an `@media (max-width: 720px)` block, and inside it (lines 1584-1589) the WCAG 2.5.5 enforcement lives:

```css
/* WCAG 2.5.5 — interactive controls ≥44×44px on touch devices */
.tab { padding: 12px 16px; min-height: 44px; }
.tab-shuffle { padding: 12px 14px; margin-left: 0; min-height: 44px; }
.tab.tab-action { padding: 12px 16px; min-height: 44px; }
.swap-btn { padding: 11px 18px; min-height: 44px; }
.btn-go { padding: 12px 22px; min-height: 44px; }
.preset { padding: 12px 14px; min-height: 44px; }
```

The comment says "on touch devices" but the gate is `max-width: 720px`. That means:

- ✅ iPhone (390px) — enforced
- ❌ **iPad portrait (768px) — not enforced** (genuinely a touch device)
- ❌ **iPad landscape (1024px) — not enforced**
- ❌ Touchscreen laptops at desktop widths — not enforced

### Recommended fix

Add (or replace with) `@media (pointer: coarse)`:

```css
@media (pointer: coarse) {
  .tab { padding: 12px 16px; min-height: 44px; }
  .tab-shuffle { padding: 12px 14px; margin-left: 0; min-height: 44px; }
  .tab.tab-action { padding: 12px 16px; min-height: 44px; }
  .swap-btn { padding: 11px 18px; min-height: 44px; }
  .btn-go { padding: 12px 22px; min-height: 44px; }
  .preset { padding: 12px 14px; min-height: 44px; }
}
```

`pointer: coarse` matches any device whose primary input is touch — regardless of viewport width. Keeps the tight desktop aesthetic for mouse users while making touch-input devices reliably tappable.

NOT applied autonomously: changing the breakpoint strategy is a design call worth user review.

### Other observations (consumer)

- `.tab` base `font-size: 0.62rem` (~10px) — at the edge of legibility on desktop. Not a regression vs. shipped state, but worth noting. The Mission Control aesthetic intentionally goes small here.
- `viewport` meta tag: `width=device-width, initial-scale=1.0` — correct.
- No horizontal scroll overflow at desktop width.
- Header height: 46px — reasonable.

## Playground findings (`/geopuesto/playground/`)

**Status**: desktop-only by inheritance from `v2-tests.html`. All 29 interactive elements measured below 44×44px CSS pixels (100% fail rate).

| Control type | Measured | Count |
|---|---|---|
| `INPUT[text]` (lat/lon entry) | 200×29 | 5 |
| `BUTTON[submit]` (Update, Set as B, etc.) | 68×29 | 14 |
| `BUTTON[submit]` (quick-pick city — Tokyo, London, etc.) | ~50×22 | most |
| `SELECT` (shape dropdown) | 100-200×~25 | 3 |
| `INPUT[range]` (spin slider, etc.) | ~200×20 | 3 |

CSS audit:
- **Zero `max-width` media queries.** No mobile breakpoint at all.
- `body { max-width: 960px }` — desktop-first.
- Quick-pick city buttons (`Tokyo`, `London`, `Sydney`, `Nairobi`, `Rio`) measure 22px tall with 11px font.

### Recommendation

The playground at maxwellhowegis.com/geopuesto/playground/ is the public research-tool URL — it shouldn't stay developer-only. Recommended pass:

1. Add a `@media (pointer: coarse)` block with `min-height: 44px` enforcement for all interactive controls.
2. Optionally widen base font-sizes from 11-12px to 14px+ for the quick-pick chips.
3. Optionally restructure the layout for narrow widths (currently desktop-only).

**Not applied autonomously** — playground/index.html is the parallel session's territory (commits `0a6c8e0` and `5d56a19`). High merge-conflict risk.

## Methodology limitations

- The Claude-in-Chrome `resize_window` is hint-only — Chrome enforces a minimum window width that prevented reaching mobile viewport sizes. CSS-pixel measurements are still accurate at any viewport.
- For true mobile testing (media queries firing, pixel-density emulation), the chrome-devtools-mcp `resize_page` CDP-based emulation would be needed — but it couldn't attach due to the user's Chrome already running on the same profile.
- This audit focuses on touch-target sizing and structural responsive design. It does NOT cover: contrast ratios, screen reader behavior, keyboard navigation patterns, color-only-conveys-information checks, or motion preferences.

## Next steps if this audit is acted on

1. Consumer: switch the `@media (max-width: 720px)` WCAG enforcement to `@media (pointer: coarse)` (single self-contained change, low risk).
2. Playground: pair with the parallel-session work — add `@media (pointer: coarse)` + a mobile layout pass when the playground's structure stabilizes.
3. Run a follow-up audit with chrome-devtools-mcp + CDP emulation to verify the fixes at real mobile viewports.
